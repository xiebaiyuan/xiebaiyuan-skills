# 工程控制论 · 工作与学习方法论

> 基于钱学森系统控制论思维框架，转化为可执行的日常工作和学习方法。
> 底层心智模型来自 [[qian-xuesen-cybernetics-thinking]]，本 Skill 聚焦「怎么用」。

## 核心隐喻

**你是一个控制系统。** 工作和学习的本质是：
- 设定目标（设定点）
- 执行动作（执行器）
- 测量结果（传感器）
- 对比目标与实际的偏差（比较器）
- 根据偏差校正下一步动作（控制器）

任何没有反馈闭环的工作都是盲飞。任何没有测量的学习都是自欺。

## 🔴 CHECKPOINT：工作检查点

| 阶段 | CHECKPOINT | 自检问题 |
|------|-----------|---------|
| 任务开始前 | 🔴 CHECKPOINT 1 | 目标可验证吗？能用一句话描述「完成标志」吗？ |
| 子任务完成后 | 🔴 CHECKPOINT 2 | 结果符合预期吗？有什么意外发现？ |
| 学习一个主题后 | 🔴 CHECKPOINT 3 | 能用自己的话给别人讲清楚吗？能回答「为什么这样设计」吗？ |
| 架构设计完成后 | 🔴 CHECKPOINT 4 | 全局最优还是局部最优？层间接口稳定吗？ |
| 技术决策做出后 | 🔴 CHECKPOINT 5 | 决策记录了吗？验证方案定了吗？复查时间设了吗？ |

**自动化场景**（cron/agent）：CHECKPOINT 跳过，结果自动投递。

## 🔴 红灯：绝对不要做

- **不要在没有闭环的情况下推进任务** — 做完不测量 = 白做
- **不要跳过定性理解直接优化** — 先理解「为什么慢」，再计算「慢多少」
- **不要追求局部精巧损害全局** — 单个模块再漂亮，增加系统复杂度就是错误
- **不要学了不练** — 学了不练 = 没学，学了不教 = 理解不深
- **不要决策不记录** — 三个月后就忘了为什么这样做

### 🔄 Fallback 路径

| 失败场景 | Fallback 方案 |
|---------|--------------|
| 任务目标无法量化 | 退而求其次：定义 3 个可观测的信号，能判断「好了/没好」 |
| 测量成本太高 | 先做最小测量：跑一个 smoke test、让一个人试用、写一段日志 |
| 学习遇到瓶颈卡住 | 换个入口：看源码卡住就看测试，看测试卡住就看 issue |
| 架构分析太复杂 | 先画最核心的 3 个组件和 2 条数据流，其余渐进补充 |
| 反馈闭环成本太高 | 先做最小闭环：加一个 assert、写一行日志、跑一个 smoke test |
| 技术决策信息不完整 | 记录「已知 / 未知 / 假设」三项，设定复查时间点 |

## 一、任务管理：反馈闭环法

### 原理
钱学森："把故障消灭在地面。" 每个任务都必须有闭环——不是做完就算，而是做完 → 测量效果 → 反馈到下一个任务。

### 执行流程

```
┌─→ 1. 定义目标 ─→ 2. 拆解子任务 ─→ 3. 执行 ─→ 4. 测量 ─┐
│                                                        │
└──────────────── 5. 偏差分析 ←──────────────────────────┘
                         │
                    6. 校正 → 进入下一个迭代
```

**Step 1: 定义目标（设定点）**
- 用一句话描述：这个任务完成的标志是什么？
- 必须可验证：「代码能跑通」比「代码写好」更精确
- 禁止模糊目标：「优化性能」→ 改为「P99 延迟从 200ms 降到 100ms」

**Step 2: 拆解子任务（层次分解）**
- 把大任务分解为 3-7 个子任务
- 每个子任务有明确的输入和输出（输入是什么，产出什么，写清楚）
- 子任务之间的依赖关系要画清楚（有依赖的串行，无依赖的并行）
- 优先做「卡脖子」的子任务——阻塞其他子任务的那个必须先做

**Step 3: 执行（执行器）**
- 一次只做一个子任务
- 每完成一个子任务，做一次微型 checkpoint：
  - 做了什么？
  - 结果符合预期吗？
  - 有什么意外发现？

**Step 4: 测量（传感器）**
- 用客观数据衡量，不用主观感觉
- 代码：跑测试、看 profiler、检查 lint
- 学习：做题、写总结、给别人讲一遍
- 文档：让别人读一遍看能不能理解

**Step 5: 偏差分析（比较器）**
- 目标 vs 实际的差距有多大？
- 偏差原因是什么？（不要归咎于「时间不够」，要找到根因）
- 这个偏差是偶发的还是系统性的？

**Step 6: 校正（控制器）**
- 系统性偏差 → 调整方法（改输入）
- 偶发偏差 → 记录为经验教训
- 偏差太大 → 重新评估目标是否合理

## 二、学习方法：综合集成法

### 原理
钱学森的「从定性到定量综合集成」：先用定性理解建立方向，再用定量方法精确化，通过人机结合迭代逼近真相。

### 执行流程

**阶段一：定性综合（建地图）**
1. 用 30 分钟快速浏览：目录、README、综述文章、Wikipedia
2. 目标：能用 3 句话向别人解释「这个东西是什么、解决什么问题、怎么解决的」
3. 画出概念地图：核心概念 + 概念之间的关系
4. 标记已知点和未知点

**阶段二：定向深入（挖隧道）**
1. 从概念地图中选一个「最值得深挖」的节点
2. 集中力量攻克这一个点（集中力量办大事）
3. 读源码、跑实验、写笔记
4. 目标：能把这个点给别人讲清楚

**阶段三：定量验证（做实验）**
1. 写代码验证理解：能跑通 ≠ 理解了，能改通 = 理解了
2. 做 benchmark：性能理解必须有数据支撑
3. 对比验证：你的理解和官方文档/源码一致吗？

**阶段四：迭代扩展（建网络）**
1. 回到概念地图，更新已知点
2. 选下一个未知点，重复阶段二三
3. 知识网络逐渐连通

### 学习检验标准

| 层次 | 检验方法 | 达标标志 | 具体做法 |
|------|---------|---------|---------|
| 知道 | 能复述 | 能用自己的话说出核心概念 | 合上文档，写 3 句话总结 |
| 理解 | 能解释 | 能回答「为什么这样设计」 | 写一段设计决策分析（为什么选 A 不选 B） |
| 应用 | 能使用 | 能用它解决实际问题 | 写一个可运行的 demo 或解决一个真实 bug |
| 分析 | 能比较 | 能说出它和替代方案的优劣 | 写一个对比表（3 个维度以上） |
| 评价 | 能判断 | 能判断什么场景该用什么不该用 | 列出 3 个适用场景 + 3 个不适用场景 |
| 创造 | 能改进 | 能提出改进建议并论证可行性 | 写一个改进建议（含预期收益和风险） |

## 三、架构设计：层次分解法

### 原理
钱学森的「层次分解与全局协调」：复杂系统具有层次结构，每层有自己的规律，层间通过明确接口协调。

### 执行流程

**Step 1: 画出系统的物理图像**
- 数据怎么流的？（输入 → 处理 → 输出）
- 组件之间怎么交互？（同步/异步/事件驱动）
- 哪些是核心路径？哪些是辅助路径？

**Step 2: 识别自然层次**
```
应用层    → 用户可见的功能
服务层    → 业务逻辑
领域层    → 领域模型和规则
基础层    → 存储、网络、计算
```
不要按技术分层，按职责分层。每一层只做自己该做的事。

**Step 3: 定义层间接口**
- 接口要稳定（变化频率低）
- 接口要窄（暴露的信息最少）
- 接口要明确（语义无歧义）

**Step 4: 验证全局最优**
- 拆完之后，系统整体复杂度是降低了还是升高了？
- 子系统之间的协调成本可控吗？
- 有没有「局部精巧但全局笨拙」的设计？

### 架构评审检查清单

```
□ 整体目标函数清晰吗？（为什么这样拆？）
□ 每层职责单一吗？（有没有一层做了两件事？）
□ 层间接口稳定吗？（改一层会不会连锁改其他层？）
□ 有反馈机制吗？（监控、日志、告警）
□ 有容错机制吗？（单点故障？降级策略？）
□ 全局最优还是局部最优？（有没有过度设计？）
```

## 四、技术决策：全局最优法

### 原理
钱学森的决策启发式：先验证后部署、全局最优压倒局部最优、自主可控优先。

### 决策流程

**Step 1: 明确决策的约束条件**
- 时间约束：有多少时间做决策？
- 信息约束：手头有多少信息？还缺什么？
- 可逆性：这个决策容易推翻吗？

**Step 2: 综合集成分析**
- 定性：团队经验、社区口碑、类似案例
- 定量：benchmark 数据、成本估算、风险概率
- 系统视角：这个决策对整体系统的影响是什么？

**Step 3: 检验决策质量**

| 检验项 | 问题 |
|--------|------|
| 全局最优 | 选这个方案，整体系统是最优的吗？还是只有这个模块最优？ |
| 自主可控 | 关键环节有没有外部依赖风险？ |
| 先验证后部署 | 有没有条件先做个小规模验证？ |
| 可逆性 | 如果选错了，推翻的成本有多高？ |
| 反馈机制 | 选了之后，怎么知道效果好不好？ |

**Step 4: 记录决策**
- 决策是什么？
- 为什么选这个？（考虑了哪些因素？排除了哪些方案？）
- 预期效果是什么？
- 怎么验证效果？
- 什么时候复查？

## 五、代码质量：反馈闭环法

### 原理
"把故障消灭在地面"——在 code review 阶段发现和修复问题，成本远低于在生产环境发现。

### 反馈闭环体系

```
写代码 → 静态检查(lint/type) → 本地测试 → Code Review → CI → 部署 → 监控
   ↑                                                              │
   └────────────────── 反馈 ──────────────────────────────────────┘
```

每一层反馈的作用：
- **Lint/Type**：即时反馈，消灭低级错误
- **本地测试**：快速反馈，验证逻辑正确性
- **Code Review**：人工反馈，发现设计问题
- **CI**：系统反馈，保证集成正确性
- **监控**：延迟反馈，发现线上问题

缺少任何一层反馈，都是在盲飞。

---

## 六、反模式

### 1. 没有闭环的任务
做完不测量 = 白做。每个任务都要有「怎么知道做完了」的标准。

### 2. 跳过定性直接定量
没有理解就开始优化 = 盲目优化。先理解「为什么慢」，再计算「慢多少」。

### 3. 追求局部精巧
一个模块写得再漂亮，如果增加了系统整体复杂度，就是错误的优化。

### 4. 没有反馈的学习
学了不练 = 没学。学了不教 = 理解不深。学了不用 = 很快就忘。

### 5. 决策不记录
做了决策但不记录原因，三个月后就忘了为什么这样做，也无法复盘。

### 6. 忽视接口设计
接口定义不清楚 = 埋雷。今天模糊一点，明天就多一个 bug。

## 使用示例

### 示例 1：接到一个新需求
**用户问**：手百扫码 SDK 要加一个新的码制支持，怎么做？

**回应**：
用反馈闭环法——
1. **定义目标**：新码制能正确解码，已有码制不受影响，性能不退化
2. **层次分解**：识别 → 定位 → 解码 → 后处理，新码制在哪个层介入？
3. **先验证后部署**：写单元测试覆盖新码制 + 回归测试覆盖已有码制
4. **反馈闭环**：CI 跑完所有测试 → 合入 → 监控线上解码成功率

### 示例 2：学习一个新的技术栈
**用户问**：想学 Compose Multiplatform，怎么学？

**回应**：
用综合集成法——
1. **定性综合**（30分钟）：看官方文档目录、README、一篇综述文章。能说出「Compose Multiplatform 是什么、解决什么问题、和 Flutter/React Native 的区别」
2. **概念地图**：UI 层 → 状态管理 → 平台适配 → 构建系统，画出依赖关系
3. **定向深入**：先从 UI 层开始，写一个 Hello World，跑通 iOS + Android
4. **定量验证**：对比两个平台的渲染性能、包体积
5. **迭代扩展**：UI 层搞通后，再搞状态管理，再搞平台适配

### 示例 3：代码质量改善
**用户问**：项目代码越来越乱，怎么改善？

**回应**：
这是系统问题，不是个人问题。用反馈闭环的思维——
1. **检查反馈机制**：有 CI 吗？有 lint 吗？有 code review 吗？有架构约束吗？
2. **定位混乱层次**：是命名不规范（微观）？模块边界不清（中观）？还是架构方向不明（宏观）？
3. **逐层治理**：先加 lint（即时反馈）→ 再加 code review（人工反馈）→ 再理清模块边界（架构反馈）
4. **"把故障消灭在地面"**：投入在 code review 上的每一分钟，都在为未来节省十倍的 debug 时间
